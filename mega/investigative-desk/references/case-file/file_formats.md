# Case File — File Formats Reference

Field-by-field reference for the files the `casefile` CLI reads and writes. All
structured files are a small, plain YAML subset (mappings, block/inline lists, quoted
strings, and folded `>` block scalars) so they stay diffable in git and parseable by
the bundled standard-library reader. See `assets/example-casefile/` for a full worked
example.

## `threads/T-XXXX.md`

One file per line of inquiry: YAML front-matter (between `---` fences) followed by a
free-text markdown body. Name files `T-0001-short-slug.md`; the numeric ID is the
join key used across the case file.

Front-matter fields:

| Field | Required | Meaning |
|---|---|---|
| `id` | yes | Canonical thread ID, e.g. `T-0001`. Defaults to the filename stem. |
| `title` | yes | One-line human title. |
| `status` | yes | One of `open`, `chasing`, `confirmed`, `cold`, `killed`. Change **only** via `update-status`. |
| `priority` | yes | Integer; lower = higher priority. Drives `brief` ordering. |
| `reason` | if cold/killed | Free-text reason the thread was parked or killed. Required by the state machine. |
| `next_action` | if open/chasing | Concrete next step. Lint flags an empty one. Cleared automatically when a thread goes cold/killed/confirmed. |
| `workflow_step` | recommended | The procedure step + script that produced the lead (provenance). |
| `entities` | optional | List of `canonical_id`s referenced; each must exist in `entities.yaml`. |
| `findings` | optional | List of finding IDs this thread produced. |
| `source_records` | if confirmed | List of source record IDs/descriptions. Required to reach `confirmed`. |

The body is free prose: hypothesis, what's documented, what's still open, dead ends.

## `entities.yaml`

The canonical entity registry. Top-level keys group entities by type (e.g. `members`,
`firms`, `clients`, `revolving_door_people`); each is a list of entries. Every entry
should carry:

| Field | Meaning |
|---|---|
| `canonical_id` | Stable ID (for members, a bioguide ID from the entity resolver). The join key. |
| `display` | Human-readable label. |
| `status` | Salience: `key`, `watch`, or `flag`. `brief` surfaces `key`/`flag`. |
| `why` | One line on why the entity matters. |

Other fields (`role`, `prior_role`, `now`, …) are free and preserved.

## `findings.yaml`

Claims that graduated from a thread to "documented from records."

```yaml
findings:
  - id: F-001
    thread: T-0001
    status: documented        # documented | confirmed | retracted
    claim: >
      Folded free-text claim...
    workflow_step: "Step 1-2, Step 5 (script.py)"
    source_records:
      - "Source record ID or description"   # >=1 required; lint flags none
    caveats: "What this does and does not establish."

legal_flags:
  - finding: F-002
    entity: some-canonical-id
    note: >
      Possible legal/ethics issue...
    status: to_verify
```

Every finding needs at least one `source_records` entry. Keep causation/influence out
of the `claim` unless the records prove it; use `caveats`.

## `journal.md`

Append-only, timestamped session log. Written only via `casefile log` and the
auto-log on `update-status`; never rewrite past entries. It is the spine of the
interaction-traces deliverable.

## `.state.json`

A derived rollup cache (thread counts by status, totals, IDs by status) regenerated by
`rollup` and after every `update-status`. Never hand-edit it; it can be git-ignored.

## The YAML subset

The bundled reader supports: mappings, block sequences (`- item`), inline flow lists
(`[a, b, c]`, including multi-line), single/double quoted strings, folded (`>`) and
literal (`|`) block scalars, `#` comments, and scalar type coercion (int/float/bool/
null). It does **not** support anchors, tags, or flow mappings — keep files within the
patterns shown in the example.
